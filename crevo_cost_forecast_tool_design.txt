Crevo Cost Forecast Tool
Design document
Date: 2026-03-29

Purpose
Design an internal tool that plugs into the Crevo backend and helps estimate:
- cloud usage
- likely monthly cost
- cost per user
- cost per workspace
- growth-based cost scenarios
- which features drive cost the most

This tool is intended for planning, pricing, and infrastructure decision-making.
It is not meant to be exact billing reconciliation software.


1. Core goal

Give the Crevo team a way to answer questions like:
- How much will Crevo likely cost at 100, 1,000, or 10,000 active users?
- What is driving cost the most right now?
- If chat usage doubles, what happens to infrastructure cost?
- If file uploads grow quickly, does Cloudinary or storage become the bottleneck?
- At what point does the current setup stop being cheap?

3. What the tool should do

Main capabilities
- track usage units from the real backend
- translate usage into cost estimates using provider pricing rules
- simulate future growth scenarios
- show monthly forecast ranges
- show cost per feature area
- identify the most expensive usage patterns

Output examples
- estimated monthly infrastructure cost
- estimated cost per active workspace
- estimated cost per monthly active user
- cost split by provider
- cost split by feature area
- optimistic / expected / high-growth scenarios


4. MVP scope

Version 1 should be rule-based and simple.

MVP should include:
- daily usage collection
- provider price configuration
- monthly cost estimator
- scenario simulator
- internal dashboard

Do not try to build:
- perfect billing reconciliation
- per-request exact infra accounting
- vendor API sync for billing on day one


5. Architecture

High-level flow

1. Backend records usage events and counters
2. Daily rollup job aggregates usage
3. Forecast service applies provider pricing formulas
4. Dashboard shows:
   - current usage
   - estimated monthly cost
   - scenario simulations
   - top cost drivers

Suggested architecture layers

Layer 1: instrumentation
- middleware and event counters

Layer 2: storage
- usage rollup tables
- pricing config tables
- forecast scenario tables

Layer 3: forecast engine
- converts usage metrics into provider cost estimates

Layer 4: admin UI
- internal dashboard for finance / ops / founder use

10. Forecast formulas for MVP

Use simple formulas first.

Examples

Requests
- monthly_requests = average_daily_requests * 30

Chat
- monthly_chat_messages = average_daily_chat_messages * 30
- estimated_chat_pressure = monthly_chat_messages + typing_events_weight

Uploads
- monthly_upload_gb = average_daily_upload_bytes * 30 / 1024^3

Emails
- monthly_emails = average_daily_emails * 30

PostHog
- monthly_events = daily_posthog_events * 30

Sentry
- monthly_sentry_events = daily_sentry_events * 30

Cost
- if flat plan:
  cost = flat monthly cost

- if per-unit:
  billable_units = max(0, usage - included_units)
  cost = billable_units * unit_price

- if thresholded:
  pick matching tier and compute

12. Security model

This tool should be admin-only.

Recommended access rules
- superAdmin only
- internal feature flag only
- no exposure to normal workspace members

Security requirements
- do not expose provider secrets in the client
- pricing config updates must be server-validated
- no raw billing credentials in the frontend


13. How to integrate with the current backend

Instrumentation points

Request middleware
- add a metrics middleware in Express
- count request totals
- count route usage
- track average duration

Service-level counters
- chat.service.ts
  - message send count
  - group/direct create count
- notification services
  - notifications created
  - stream connections
- asset.service.ts
  - upload count
  - bytes uploaded
- email.service.ts
  - email count by type

Daily aggregation job
- run once per day
- write into usage_daily_rollups

Suggested implementation order
1. request metrics middleware
2. event counters in major services
3. daily rollup persistence
4. forecast service
5. admin UI

15. Benefits

Business benefits
- helps forecast infrastructure spend early
- helps decide pricing
- helps identify expensive features before they become a problem
- reduces surprise bills

Product benefits
- shows which usage patterns are worth optimizing
- helps prioritize engineering work
- supports growth planning


16. Tradeoffs

What this tool will do well
- directional cost forecasting
- scenario planning
- identifying top cost drivers

What it will not do perfectly
- exact vendor invoice matching
- precise infra micro-cost accounting
- perfect prediction under big architecture changes


17. Recommended MVP conclusion

Yes, this tool is worth building.

Best first version:
- backend usage instrumentation
- daily rollup table
- provider pricing config
- admin-only forecast dashboard
- scenario simulator

That version will already be valuable for planning and pricing.


18. Suggested future upgrades

Later improvements
- vendor billing API sync
- usage anomaly alerts

- AI cost simulation

19. Final product principle

This tool should help answer:
- how much are we spending?
- what is making it expensive?
- what happens if usage grows?
- where should we optimize first?

It should be simple enough to trust and practical enough to use regularly.
